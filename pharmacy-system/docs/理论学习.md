项目核心请求链路：

```text

        ↓

        ↓

        ↓

        ↓

        ↓

        ↓

        ↓

        ↓
数据库返回数据
        ↓
后端通过 Result 封装返回
        ↓
前端拿到数据并展示到页面
```

## 3.1 API 是什么

API 可以理解为：

```text
前端找后端办事的地址。
```

例如：

```text
/api/client/chat       AI 客服接口
/api/client/medicines  用户端药品查询接口

```

前端要查询药品，就请求药品查询 API。

前端要发送 AI 客服问题，就请求 AI 客服 API。

常见请求方法：

```text

```

例子：


## 3.3 JSON 是什么

JSON 是前后端传数据常用格式。

请求示例：

```json

}
```

返回示例：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "reply": "本店营业时间为 08:00 - 21:30。"
  }
}
```

记忆方式：

```text
JSON = 前后端传数据的一种格式，看起来像 JavaScript 对象。
```

## 3.4 axios 是什么

axios 是前端用来发送 HTTP 请求的工具。

在项目里，axios 被统一封装在：

```text
pharmacy-client/src/utils/request.js
pharmacy-ui/src/utils/request.js
```

作用：

```text
统一配置 baseURL
统一配置 timeout
统一发送请求
后续可以统一处理 token、错误提示等
```

## 3.5 request.js 为什么配置 baseURL

用户端 `request.js` 中有：

```js
const request = axios.create({
  baseURL: '/api',
  timeout: 10000
})
```

意思是：

```text
所有通过 request 发出的请求，都会自动加上 /api。
```

例如 api 文件写：

```text
/client/medicines
```

实际请求变成：

```text
/api/client/medicines
```

## 3.6 Vite proxy 是什么

开发环境下：

```text
后端运行在 8080
管理端前端运行在 3000
用户端前端运行在 5174
```

前端要访问后端，需要代理。

用户端 `pharmacy-client/vite.config.js` 中有：

```js
proxy: {
  '/api': {
    target: 'http://127.0.0.1:8080',
    changeOrigin: true
  }
}
```

意思是：

```text
前端开发服务器收到 /api 开头的请求，会转发给后端 8080。
```

所以浏览器看到：

```text
http://127.0.0.1:5174/api/client/medicines
```

真正后端处理：

```text
http://127.0.0.1:8080/api/client/medicines
```

```text
3000 是管理端前端页面端口；
```


## 3.9 示例二：药品查询请求链路

涉及文件：

```text
前端页面：
pharmacy-client/src/views/medicines/index.vue

前端 API：
pharmacy-client/src/api/client-medicine.js

axios 封装：
pharmacy-client/src/utils/request.js

Vite 代理：
pharmacy-client/vite.config.js

后端 Controller：
pharmacy-admin/src/main/java/com/pharmacy/controller/ClientMedicineController.java

后端 Mapper：
pharmacy-admin/src/main/java/com/pharmacy/mapper/MedicineMapper.java

返回对象：
pharmacy-admin/src/main/java/com/pharmacy/vo/ClientMedicineVO.java
```

完整流程：

```text
用户在 /medicines 输入“布洛芬”
        ↓
点击搜索按钮
        ↓
views/medicines/index.vue 触发 handleSearch()
        ↓
handleSearch 调用 loadMedicines()
        ↓
loadMedicines 调用 getClientMedicines(buildParams())
        ↓
client-medicine.js 发送 GET /client/medicines
        ↓
request.js 自动加上 baseURL: /api
        ↓
实际请求变成 GET /api/client/medicines
        ↓
vite.config.js 把 /api 请求代理到后端 8080
        ↓
ClientMedicineController 中的 list 方法接收请求
        ↓
@RequestParam 接收 current、size、keyword 等参数
        ↓
Controller 调用 MedicineMapper.selectClientMedicinePage(...)
        ↓
MedicineMapper 查询 medicine 表和 stock 表
        ↓
返回 Page<ClientMedicineVO>
        ↓
Controller 用 Result.success(result) 封装
        ↓

        ↓

```

## 3.10 第二阶段答辩表达

如果老师问：

```text
用户端药品查询是怎么从前端请求到后端的？
```

可以回答：

```text
用户在药品查询页面输入关键词并点击搜索后，页面会触发 handleSearch 方法，进而调用 loadMedicines 方法。loadMedicines 会通过 buildParams 组装 current、size、keyword、type、dosageForm、isRx 和 stockStatus 等查询参数，然后调用 client-medicine.js 中封装的 getClientMedicines 方法。

getClientMedicines 使用 request，也就是封装后的 axios，发送 GET 请求。因为 request.js 中配置了 baseURL 为 /api，所以请求地址会从 /client/medicines 拼接成 /api/client/medicines。

开发环境下，Vite 会通过 proxy 把 /api 开头的请求转发到 Spring Boot 后端 8080 端口。后端 ClientMedicineController 通过 @GetMapping 接收请求，使用 @RequestParam 获取分页和筛选参数，然后调用 MedicineMapper 查询 medicine 和 stock 表，返回 Page<ClientMedicineVO>，最后通过 Result.success 统一封装后返回给前端展示。
```

## 4.1 Controller 是什么

Controller 是后端入口。

职责：

```text
接收前端请求
接收请求参数
调用 Service
返回结果给前端
```

答辩表达：

```text
Controller 是控制层，主要负责接收前端 HTTP 请求，进行参数接收，然后调用 Service 层完成业务处理，最后通过统一 Result 返回给前端。
```

注意：

```text
Controller 不应该写太多复杂业务逻辑。
```

## 4.2 Service 是什么

Service 是业务逻辑层。

职责：

```text
处理业务规则
判断业务状态
协调多个表
调用 Mapper
控制事务
决定成功或失败
```

例如销售出库，Service 要处理：

```text
查询销售订单是否存在
判断订单是否已经出库
判断订单是否已退款
查询销售明细
判断库存是否足够
扣减库存
更新销售订单状态
必要时更新会员累计消费
任何一步失败都要回滚
```

答辩表达：

```text
Service 是业务逻辑层，负责处理具体业务规则。例如销售出库时，需要校验订单状态、判断库存是否足够、扣减库存并更新订单状态，这些逻辑都应该放在 Service 层。
```

## 4.3 Mapper 是什么

Mapper 是数据访问层。

职责：

```text
查询数据库
新增数据
修改数据
删除数据
执行 SQL
```

例如：

```text
MedicineMapper 操作药品表
StockMapper 操作库存表
SaleOrderMapper 操作销售订单表
PurchaseOrderMapper 操作采购订单表
```

答辩表达：

```text
Mapper 是数据访问层，负责和数据库交互。项目中使用 MyBatis-Plus，简单 CRUD 可以通过 BaseMapper 完成，复杂查询则可以通过自定义 SQL 或注解 SQL 实现。
```

## 4.4 Entity 是什么

Entity 是数据库实体类，通常和数据库表对应。

例如：

```text
Medicine.java        对应 medicine 药品表
Stock.java           对应 stock 库存表
SaleOrder.java       对应 sale_order 销售订单表
PurchaseOrder.java   对应 purchase_order 采购订单表
Customer.java        对应 customer 顾客/会员表
```

答辩表达：

```text
Entity 是实体类，主要和数据库表结构对应，用来承载数据库中的一条记录。
```

## 4.5 DTO 是什么

DTO 用于接收前端传入的请求参数。

可以记：

```text
DTO = 进来的数据
```

例如：

```text
ClientChatDTO          接收 AI 客服 message
SaleOrderCreateDTO     接收创建销售订单所需参数
PurchaseOrderCreateDTO 接收创建采购订单所需参数
```

答辩表达：

```text
DTO 用于接收前端传入的请求参数，尤其是创建订单这种包含多个字段和明细列表的复杂请求，使用 DTO 可以让参数结构更清晰。
```

## 4.6 VO 是什么

VO 用于返回给前端展示。

可以记：

```text
VO = 出去的数据
```

例如：

```text
ClientMedicineVO           用户端药品展示对象
ClientStoreInfoVO          用户端药店信息展示对象
ClientMemberProfileVO      用户端会员资料展示对象
ClientMemberOrderVO        用户端会员订单展示对象
ImportResultVO             Excel 导入结果展示对象
```

答辩表达：

```text
VO 用于封装返回给前端展示的数据。它不一定和数据库表完全一致，可以根据页面展示需求组合多个表的数据。
```

## 4.7 DTO 和 VO 的区别

核心记忆：

```text
DTO 是前端传给后端的数据；
VO 是后端返回给前端看的数据。
```

再简单一点：

```text
DTO 是进来的；
VO 是出去的。
```

## 4.8 Result 是什么

Result 是统一返回结果封装。

格式示例：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

作用：

```text
统一后端返回格式
方便前端判断成功或失败
方便统一错误提示
```

答辩表达：

```text
Result 是统一响应结果封装，保证后端接口返回格式一致，前端可以统一处理成功和失败情况。
```

## 4.9 @Transactional 是什么

`@Transactional` 表示事务控制。

适用场景：

```text
一次业务涉及多个数据库操作，必须同时成功或同时失败。
```

例如销售出库涉及：

```text
扣减库存
更新订单状态
更新会员累计消费
```

如果扣了库存，但订单状态更新失败，数据就不一致。

所以要用事务：

```text
要么全部成功，要么全部失败回滚。
```

答辩表达：

```text
销售出库属于多表联动业务，所以我把核心逻辑放在 Service 层，并使用 @Transactional 保证事务一致性。如果库存扣减或订单状态更新过程中任何一步失败，事务会回滚，避免出现库存和订单状态不一致的问题。
```

## 4.10 为什么不能全写在 Controller

如果所有逻辑都写在 Controller：

```text
Controller 会非常长
业务逻辑混乱
不好维护
不好复用
不好测试
后续功能多了很难扩展
```

分层之后：

```text
Controller 只管请求
Service 只管业务
Mapper 只管数据库
```

答辩表达：

```text
如果把所有逻辑都写在 Controller 中，会导致代码耦合严重，不方便维护。分层之后，Controller、Service、Mapper 各自负责不同职责，代码结构更清晰，也方便后续扩展。
```

## 4.11 后端分层核心背诵版

```text
Controller：接收请求，调用 Service，返回结果。
Service：处理业务逻辑，比如库存校验、订单状态判断、采购入库、销售出库。
Mapper：操作数据库，执行增删改查。
Entity：数据库表对应的 Java 类。
DTO：前端传给后端的数据。
VO：后端返回给前端展示的数据。
Result：统一返回格式。
```

```text
我的后端采用 Spring Boot 分层架构，主要分为 Controller、Service、Mapper、Entity、DTO、VO 等层。

Controller 层负责接收前端请求，例如药品管理、采购订单、销售订单、库存管理和用户端查询等接口都会先进入 Controller。

Service 层负责业务逻辑，比如采购入库时自动增加库存，销售出库时校验库存并扣减库存，这些涉及多表联动和事务控制的逻辑都放在 Service 层。

Mapper 层负责数据库访问，项目使用 MyBatis-Plus，简单 CRUD 通过 BaseMapper 完成，复杂查询通过自定义 SQL 完成。

Entity 用来对应数据库表，DTO 用来接收前端请求参数，VO 用来封装返回给前端展示的数据，Result 用来统一接口返回格式。

这样分层的好处是职责清晰，Controller 不直接写复杂业务，Mapper 不处理业务规则，后续维护和扩展更加方便。
```

## 5.2 main.js 是什么

```text
main.js 是 Vue 项目的入口文件，负责创建应用实例、注册路由和组件库，并将应用挂载到页面上。
```

## 5.3 App.vue 是什么

`App.vue` 是根组件，像整个前端的外壳。

```text
RouterView
```

意思是：

```text
router 用于配置前端路由，建立访问路径和页面组件之间的对应关系。
```

## 6. 项目 3 分钟概述

用于毕业答辩或面试开场。

```text
我的项目是乡镇药店管理系统，主要面向乡镇单体药店和小型药房的日常经营管理场景。项目采用前后端分离架构，目标是解决传统药店依赖纸质台账或简单表格管理时，容易出现药品信息分散、库存不清、采购销售记录难追踪、会员消费记录不完整、操作过程不可审计等问题。

整个系统分为三个核心工程。第一个是 pharmacy-admin，也就是后端服务模块，基于 Spring Boot 3.2.5、MyBatis-Plus、Spring Security、Spring AOP、EasyExcel 和 MySQL 开发，运行环境使用 JDK 17。后端主要负责提供 REST API、处理业务逻辑、访问数据库、统一返回结果，以及完成库存联动、Excel 导入导出和操作日志等功能。

第二个是 pharmacy-ui，也就是管理端前端，基于 Vue 3、Vue Router、Element Plus、Axios 和 Vite 开发，主要面向药店工作人员或管理员使用。管理端包含登录、控制台、药品管理、供应商管理、顾客/会员管理、采购订单、采购明细、销售订单、销售明细、库存管理、库存盘点、用户管理和系统日志等页面。工作人员可以通过页面完成药品档案维护、供应商维护、会员维护、采购录入、销售开单、库存查看、盘点记录和日志追踪。

第三个是 pharmacy-client，也就是用户端前端，同样基于 Vue 3 开发，主要面向普通顾客。用户端包含首页、药品查询、药品详情、会员中心、药店信息和 AI 客服。顾客可以查询药品信息、查看库存状态、了解药店营业时间、地址和联系电话，也可以通过会员中心查看自己的资料和消费记录。

项目后端目前已经经历了两次比较明显的优化补充。第一次主要补充了用户端功能和规则型 AI 客服。AI 客服没有接入外部大模型，而是通过后端 ClientChatController 做关键词规则匹配，可以回答营业时间、药店地址、联系电话、处方药购买说明、会员消费记录入口和药品查询入口等常见问题。对于具体病情、诊断、剂量和联合用药问题，系统会做安全兜底，提示用户咨询医生或药师。

第二次优化主要增强了管理端业务闭环。后端新增和完善了采购入库、销售出库与库存自动联动逻辑。采购订单确认入库后，系统会根据采购明细自动增加库存，并且防止重复入库；销售订单确认出库后，系统会校验库存是否足够，库存充足时自动扣减库存，库存不足时返回错误提示，同时也会拦截重复出库和异常状态出库。相关逻辑放在 PurchaseOrderServiceImpl 和 SaleOrderServiceImpl 中，并使用 @Transactional 保证订单状态和库存变化同时成功或失败。

另外，项目还补充了 Excel 导入导出能力。后端引入 EasyExcel，支持药品数据导出、药品导入模板下载、药品导入、库存数据导出、采购订单导出和销售订单导出。后端也新增了 excel 相关行模型、ExcelResponseUtil 工具类以及 ImportResultVO 返回导入结果，方便前端展示成功数量、失败数量和错误信息。

后端整体按照 Controller、Service、Mapper、Entity、DTO、VO 分层组织。Controller 负责接收前端请求，Service 负责业务逻辑，Mapper 负责数据库访问，Entity 对应数据库表，DTO 用于接收前端请求参数，VO 用于返回前端展示数据。

数据库方面，系统使用 MySQL，核心表包括用户表、药品表、供应商表、顾客/会员表、采购订单表、采购明细表、销售订单表、销售明细表、库存表、库存盘点表和系统日志表。当前库存预警下限已经放在 stock 表中，用于按批次维护库存预警。数据库连接信息通过 DB_URL、DB_USERNAME、DB_PASSWORD 环境变量配置，避免把真实密码写入代码。

部署方面，项目计划采用 Nginx 静态资源托管前端，再通过 Nginx 反向代理把 /api 请求转发到 Spring Boot 后端。开发环境下，后端默认运行在 8080 端口，管理端前端运行在 3000 端口，用户端前端运行在 5174 端口，两个前端都通过 Vite proxy 把 /api 请求代理到后端。

总的来说，这个项目从最初的基础药店管理系统，已经扩展成包含管理端、用户端、AI 客服、库存联动和 Excel 导入导出的完整前后端分离项目，既覆盖了药店日常进销存业务，也具备一定的展示性和扩展性。
```
